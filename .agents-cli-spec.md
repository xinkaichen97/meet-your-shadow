# Agent Spec

## Overview
Shadow Self-Reflection Agent: a single-session ADK app. The user answers 16
Likert-scale questions client-side (paired direct/projection items across 8
Jungian "shadow types"). The backend scores tension between each pair,
optionally asks one clarifying follow-up, then an LLM writes a personalized
300-500 word narrative. After the report, the user can continue in free text.
Full design lives in `shadow_test_agent_spec.md` (project root) — this file
tracks scope/build decisions on top of it.

## Example Use Cases
- User submits 16 answers -> Orchestrator calls AnalystTool -> tension scores
  computed -> if `followup_needed`, FollowUpTool returns the two-option
  clarifying question; otherwise InterpreterTool generates the report.
- User answers the follow-up -> InterpreterTool generates the report using
  top_type + answers + follow-up choice.
- After the report, user sends free text -> InterpreterTool responds in
  "conversation" mode, grounded in the existing report.

## Tools Required
- `analyst_tool` (FunctionTool, deterministic) — scores all 16 answers,
  writes `tension_scores` / `top_type` / `followup_needed` to session state.
- `followup_tool` (FunctionTool, deterministic) — fixed template lookup for
  the top shadow type's clarifying question (8 hardcoded templates).
- `interpreter_tool` (AgentTool wrapping an LlmAgent) — the only open-ended
  LLM generation: initial narrative report, and later free-text conversation
  grounded in that report. No external APIs.
- `record_answers` (plain function, not agent-wired) — writes a submitted
  answer batch into session state; owned by the future FastAPI layer, not
  called by the Orchestrator.

## Constraints & Safety Rules
- Orchestrator never generates narrative/interpretation itself — only
  decides which tool to run and bridges with 1-2 short sentences.
- InterpreterTool: never diagnostic/clinical language, never a numbered list
  of fixes, never says "you have a problem," ends on acknowledgment not
  resolution, never invents details the user didn't provide.
- Crisis-keyword detection is a plain-Python check that must run before the
  Runner is ever invoked and must never reach the LLM — belongs to the
  FastAPI layer (deferred, see Scope Decisions).

## Success Criteria
- `agents-cli run` smoke tests: full 16-answer submission correctly routes
  through AnalystTool -> (FollowUpTool if triggered) -> InterpreterTool, and
  produces a report matching the tone constraints above.
- A second free-text turn after the report is answered in "conversation"
  mode without regenerating or repeating the report.

## Reference Samples
None of the catalog samples match (not RAG, not scheduled/ambient, not
OAuth, not cross-session memory) — built from scratch on the base `adk`
template.

## Scope Decisions (asked and answered before scaffolding)
- **MCP tool for Jungian reference material**: skipped for this build.
  InterpreterTool relies solely on the anchor table in
  `shadow_test_agent_spec.md` section 6.
- **Deployment target**: prototype only (`--prototype`, no deployment
  target). Test locally via `adk api_server` / `agents-cli run`. Add Cloud
  Run later with `agents-cli scaffold enhance . --deployment-target
  cloud_run` when ready to deploy.
- **Build scope**: ADK agent/tool definitions only in this pass (Orchestrator,
  AnalystTool, FollowUpTool, InterpreterTool). The FastAPI wrapper
  (`record_answers` call site, crisis-keyword short-circuit, `/run_sse`
  session wiring per spec section 9-11) is a follow-up step, not built yet.
- **Session storage**: `in_memory` (single-session prototype; no persistence
  requirement stated).
